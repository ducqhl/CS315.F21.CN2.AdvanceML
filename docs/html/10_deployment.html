mongodb  (independent — no dependencies)
  ├── api        (depends_on: mongodb — condition: service_started)
  ├── dashboard  (depends_on: mongodb — condition: service_started)
  └── inference  (depends_on: mongodb — condition: service_started)

spark-master  (independent)
  └── spark-worker  (depends_on: spark-master — condition: service_healthy)

api
  └── frontend  (depends_on: api — condition: service_started)
Health check conditions: condition: service_healthy đảm bảo service phụ thuộc chỉ start sau khi health check pass. Tránh race condition như Producer start trước khi Kafka sẵn sàng nhận connections — điều này gây Producer crash ngay khi startup.

Docker Volumes

VolumeMount Point (container)Dùng choPersistence
mongodb_data /data/db MongoDB data — không mất khi container restart Named volume, persistent
./src /app/src Source code bind mount — hot reload trong dev Bind mount
./data /app/data CSV lịch sử + model artifacts (lstm_*.pt, scaler_*.pkl) Bind mount
Kafka checkpoint (/tmp): Spark checkpoint dir là /tmp/spark-checkpoints bên trong container — mất khi container restart. Sau restart, Spark đọc từ latest Kafka offset, bỏ qua data trong downtime. Production cần mount persistent volume cho checkpoint dir (HDFS, S3, hoặc named volume).

Tại sao Docker Compose thay vì Kubernetes?

Docker Compose (chọn)Kubernetes
Single-machine deployment — phù hợp với laptop/server local Multi-node cluster management — overkill cho single machine
Đơn giản: 1 file YAML, make docker-up Phức tạp: manifests, services, ingress, PVC, secrets management
Learning curve thấp — không che khuất Lambda Architecture concepts Learning curve cao — K8s complexity che khuất mục tiêu học tập chính
Migration path: Docker Compose → Docker Swarm → Kubernetes Phù hợp cho production scale, multi-datacenter
