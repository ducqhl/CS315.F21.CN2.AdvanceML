Pipeline Tổng quan

Speed Layer sử dụng Spark Structured Streaming với hai parallel queries. Query A tính windowed aggregations (20-min tumbling windows, 5-min slide). Query B tính per-record technical indicators (RSI, Bollinger, VWAP, ATR). Cả hai ghi vào MongoDB qua pattern foreachBatch.

Query A — Window Aggregation

Output: window_stats

Type: Stateful — groupBy + sliding window (20min/5min)

State: Giữ window state trong memory cho đến khi watermark pass

Checkpoint: Lớn hơn (offset + aggregation state)

Query B — Per-Record Enrichment

Output: realtime_prices + alerts

Type: Stateless per-batch — PySpark Window Functions

State: Không giữ state giữa các micro-batch

Checkpoint: Nhỏ hơn (chỉ offset)

Bước 1 — Đọc Stream từ Kafka

df_raw = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", KAFKA_BOOTSTRAP)   # kafka:29092 (internal)
    .option("subscribe", KAFKA_TOPIC_RAW)                  # crypto_raw
    .option("startingOffsets", "latest")
    .option("maxOffsetsPerTrigger", "1000")
    .option("failOnDataLoss", "false")
    .load()
failOnDataLoss=false: Nếu Kafka topic bị xóa hoặc offset bị reset (retention policy), Spark không crash — tiếp tục từ earliest available. Phù hợp cho demo setup. Production cần true nếu at-least-once delivery là bắt buộc.

Bước 2 — Parse JSON và Watermark

# Parse value bytes → JSON → typed DataFrame
parsed_df = df_raw.select(
    F.from_json(F.col("value").cast("string"), CRYPTO_SCHEMA).alias("data"),
    F.col("timestamp").alias("kafka_ts")
).select("data.*", "kafka_ts")

# Convert event_time string → TimestampType (UTC)
parsed_df = parsed_df.withColumn(
    "event_time",
    F.to_timestamp(F.col("timestamp"), "yyyy-MM-dd'T'HH:mm:ssXXX")
)

# Watermark: tolerate data late up to 10 minutes
parsed_df = parsed_df.withWatermark("event_time", "10 minutes")
Watermark = 10 phút — bằng đúng 1 poll cycle của Producer (600s), tránh bỏ sót data hợp lệ
❌ Không có watermark ✅ Với watermark 10 phút
Spark giữ state cho tất cả windows chưa đóng → RAM tăng không giới hạn → OOM sau vài ngày Spark chỉ giữ state cho windows trong vòng 10 phút qua event_time. State được released sau khi watermark vượt qua.
Data trễ luôn được xử lý (không sustainable với long-running job) Data trễ hơn 10 phút bị discard. Acceptable vì producer poll mỗi 10 phút — data không thể trễ hơn 1 poll cycle.

Query A — Window Aggregation

Tính OHLCV rollups theo sliding window 20 phút / 5 phút slide. Kết quả ghi vào window_stats.

windowed = parsed_df \
    .groupBy(
        F.window("event_time", "20 minutes", "5 minutes"),  # 20-min window, 5-min slide
        "coin"
    ) \
    .agg(
        F.avg("price_usd").alias("sma_20"),
        F.max("price_usd").alias("high_window"),
        F.min("price_usd").alias("low_window"),
        F.sum("volume_24h").alias("total_volume"),
        F.last("market_cap").alias("market_cap"),
    )

windowed.writeStream \
    .trigger(processingTime="30 seconds") \
    .outputMode("append") \
    .foreachBatch(lambda df, id: write_batch(df, "window_stats", batch_id=id)) \
    .option("checkpointLocation", f"{CHECKPOINT_DIR}/window_stats") \
    .start()
Sliding vs Tumbling window: Window 20 phút slide mỗi 5 phút → 1 event thuộc tối đa 4 windows chồng lên nhau. Cho SMA-20 mượt hơn tumbling window (không có biên cứng). outputMode("append") phù hợp vì với watermark, mỗi window chỉ được output sau khi đóng.

foreachBatch Pattern — Exactly-Once Writes

def write_batch(batch_df: DataFrame, collection_name: str, batch_id: int):
    """
    Idempotent write: delete existing records for this batch_id,
    then insert new ones. Safe to retry.
    """
    records = batch_df.toPandas().to_dict("records")
    if not records:
        return

    col = db[collection_name]
    # Step 1: Remove any previous attempt with same batch_id (idempotent)
    col.delete_many({"batch_id": batch_id})
    # Step 2: Insert current batch
    for r in records:
        r["batch_id"] = batch_id
        r["created_at"] = datetime.utcnow()
    col.insert_many(records)

Idempotency guarantee

Chạy lại cùng batch_id → cùng kết quả. delete_many(batch_id) trước insert_many loại bỏ duplicates từ Spark retry.

Tại sao không dùng Connector?

com.mongodb.spark không ổn định trong streaming mode — checkpoint state bị corrupt sau restart. foreachBatch + PyMongo cho full control.

Tại sao foreachBatch thay vì MongoDB Spark Connector? MongoDB Spark Connector (com.mongodb.spark) có vấn đề về checkpointing trong streaming mode — không ổn định với Spark Structured Streaming. foreachBatch gọi PyMongo trực tiếp, cho phép exactly-once semantics thông qua delete+insert với batch_id. Pattern này đảm bảo: nếu micro-batch bị retry (Spark failure), delete_many trước khi insert loại bỏ duplicates.

Checkpoint & Fault Tolerance

⚠️ Production gap: Checkpoint ở /tmp — mất khi container restart. Production cần persistent volume (HDFS, S3, Docker named volume).
QueryCheckpoint DirNội dung được lưu
Query A (window_stats) /tmp/spark-checkpoints/window_stats Lưu Kafka offset đã xử lý + window aggregation state. Khi restart, tiếp tục từ offset cũ mà không reprocess.
Query B (enrichment) /tmp/spark-checkpoints/enrichment Lưu offset. Per-record enrichment không có stateful aggregation nên checkpoint nhỏ hơn Query A.
Checkpoint ở /tmp: Mất khi container restart. Sau restart, Spark đọc lại từ startingOffsets="latest" — bỏ qua data trong thời gian downtime. Trong production, checkpoint nên ở persistent storage (HDFS, S3, hoặc Docker volume mount). Đây là trade-off cho simplicity trong demo setup.

Các chỉ số kỹ thuật — Công thức chi tiết

Chỉ sốLoạiCông thứcÝ nghĩa thực tiễn
SMA-n Trend \(\displaystyle\frac{1}{n}\sum_{i=0}^{n-1} p_{t-i}\) Trung bình động n kỳ — lọc noise, xác định xu hướng ngắn/trung hạn
RSI-14 Momentum \(100 - \dfrac{100}{1 + \bar{U}_{14}/\bar{D}_{14}}\) Sức mạnh tương đối [0–100]. >70: overbought (có thể sắp giảm), <30: oversold (có thể sắp tăng)
Bollinger Bands Volatility \(\text{SMA}_{20} \pm 2\sigma_{20}\) Dải giá ±2 std. Giá vượt band upper/lower = breakout signal. Band giãn nở = volatility tăng
VWAP Volume \(\displaystyle\frac{\sum_i p_i\, v_i}{\sum_i v_i}\) Giá trung bình có trọng số theo khối lượng — benchmark cho institutional traders
ATR-14 Volatility \(\displaystyle\frac{1}{14}\sum_{i=1}^{14} TR_i\) Biên độ dao động thực trung bình 14 kỳ — đo volatility không phân biệt hướng tăng/giảm
RSI warmup limitation: RSI-14 cần đủ 14 data points để chính xác. Trong streaming mode, các record đầu tiên của session (ít data trong window) RSI sẽ không chính xác. Đây là inherent limitation của streaming calculation — batch layer tính RSI chính xác trên full history, streaming RSI là approximate cho realtime display.
