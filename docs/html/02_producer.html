CoinGecko Producer Speed Layer — điểm đầu vào dữ liệu · src/producer/crypto_producer.py

2 coins (BTC + DOGE) 600s poll interval 12 fields/message 7.200 calls/month acks=all reliability 2 write destinations

Trách nhiệm

Producer là điểm đầu vào duy nhất của dữ liệu thị trường vào hệ thống. Nó thực hiện ba việc đồng thời:

1. Thu thập dữ liệu

Gọi CoinGecko API mỗi 600 giây để lấy giá tức thời (BTC, DOGE) và OHLC 4 giờ mỗi 3 chu kỳ (mỗi 30 phút).

2. Publish lên Kafka

Serialize thành JSON message và publish lên Kafka topic crypto_raw với key là coin symbol. Cấu hình acks=all, retries=3.

3. Dual-write tới MongoDB

Song song ghi raw tick vào live_prices collection (không qua Spark). API có thể đọc raw tick ngay lập tức mà không đợi Spark process.

Kafka Message Schema (12 trường)

Mỗi message được publish lên topic crypto_raw có cấu trúc JSON đầy đủ sau:

{
  "coin":       "BTC",                           // trading symbol
  "coin_id":    "bitcoin",                       // CoinGecko internal ID
  "price_usd":  77473.37,                        // giá hiện tại (USD)
  "volume_24h": 32816206284.10,                  // khối lượng giao dịch 24h
  "market_cap": 1548812855182.45,                // vốn hóa thị trường
  "change_24h": -1.24,                           // % thay đổi trong 24h
  "timestamp":  "2026-06-02T10:30:00+00:00",    // ISO-8601 UTC
  "source":     "coingecko",
  "open":       77200.00,   // từ OHLC 4h candle gần nhất (nullable)
  "high":       77600.00,
  "low":        77100.00,
  "close":      77473.37
}
OHLC nullable: Các trường open/high/low/close là null trong 2 chu kỳ đầu (OHLC chỉ fetch mỗi 3 chu kỳ, tức mỗi 30 phút). Spark Streaming xử lý null này bình thường vì OHLC_SCHEMA định nghĩa các field là nullable=True.

Rate Limiting Strategy

CoinGecko demo tier giới hạn 10.000 calls/tháng. Budget được tính chặt chẽ để tránh exceed quota:

Budget Calculation

Poll interval = 600s → 6 cycles/giờ → 4.320 price calls/tháng. OHLC fetch mỗi 3 cycles × 2 coins → 2.880 OHLC calls/tháng.

Price calls (/simple/price)4.320 / 10.000
OHLC calls (/coins/{id}/ohlc)2.880 / 10.000
Total sử dụng7.200 / 10.000 (72%)
Dự phòng: 2.800 calls (28%) Buffer dùng cho: HTTP 429 retries Max coins khả dụng: +1 coin (~3.600 calls thêm)
Rate limit HTTP 429: Khi CoinGecko trả về 429, producer tự động sleep thêm 60 giây trước khi thử lại. Logic này nằm trong produce_loop() với except requests.exceptions.HTTPError as e: if e.response.status_code == 429. Không tính là consecutive error.

Error Handling & Exponential Backoff

HTTP 429 — Rate Limited

Sleep thêm 60 giây. Không tăng consecutive_errors. Sau đó tiếp tục normal cycle.

Lỗi khác (network, timeout, parse error...)

Tăng consecutive_errors. Backoff = min(30 × 2^(n-1), 3600) giây.

consecutive_errors = 1 → backoff = 30s
consecutive_errors = 2 → backoff = 60s
consecutive_errors = 3 → backoff = 120s
consecutive_errors = 4 → backoff = 240s
...
consecutive_errors = 7 → backoff = 3600s (max 1 giờ)

Reset sau thành công

Sau mỗi chu kỳ thành công, consecutive_errors = 0. Backoff trở về 0. Cycle tiếp tục với poll interval 600s bình thường.

OHLC Fetching Logic

# cycle = 0, 1, 2, 3, 4, 5, ...
# OHLC_POLL_MULTIPLIER = 3  (config)

fetch_ohlc_this_cycle = (cycle % OHLC_POLL_MULTIPLIER == 0)
# True khi cycle = 0, 3, 6, 9, ... (mỗi 1800 giây = 30 phút)

if fetch_ohlc_this_cycle:
    ohlc_data = coingecko.get_coin_ohlc_by_id(coin_id, vs_currency='usd', days=1)
    # Lấy candle gần nhất (phần tử cuối của list)
    latest_candle = ohlc_data[-1]  # [timestamp, open, high, low, close]
    message["open"]  = latest_candle[1]
    message["high"]  = latest_candle[2]
    message["low"]   = latest_candle[3]
    message["close"] = latest_candle[4]
else:
    # Các trường OHLC = null trong 2 chu kỳ không fetch
    message["open"] = message["high"] = message["low"] = message["close"] = None

Khi OHLC fetch thành công, candle gần nhất (phần tử cuối của list) được lấy làm open/high/low/close cho message. Nếu OHLC fetch fail, các trường này là null — không làm gián đoạn price feed.

Design principle: Producer được thiết kế để không bao giờ stop. Mọi lỗi đều được handle với backoff hoặc skip. Main loop chỉ dừng khi có KeyboardInterrupt. Đây là pattern phù hợp cho long-running service trong Docker container.
